# TS Manim Three-Style Interaction Plan

## Goal

Add a Three.js-like interaction layer to the TS Manim API so rendered
objects can participate in picking, hover, selection, and dragging
without breaking the current scene/timeline model.

The first architectural move is to drop support for non-Three.js TS
Manim rendering. This plan assumes the TS runtime converges on a single
Three.js-backed renderer rather than preserving SVG and Three render
paths in parallel.

This plan does not implement the feature. It defines the target
architecture, API shape, identity rules, migration phases, and
validation work.

## Main Constraint

Every interactive renderable object must have a stable id that encodes
the file and line of the Manim source that created the corresponding
`Mobject`.

This is a hard requirement for all new interaction work.

Examples of acceptable id shapes:

- `py/01_mobjects_basics.py:42:square_1`
- `app/src/lib/ts-feature-sweep/ts/mobjectsBasics.ts:118:title_box`
- `py/19_geometry_and_text_primitives.py:87:auto_3`

The exact encoding format can change, but the id must always preserve:

- repo-relative source file
- 1-based source line
- local discriminator or symbolic suffix when needed

## Why This Matters

Three.js-style interaction depends on stable identity.

Today the TS Manim side is built around:

- scene-data-first `Mobject` instances
- snapshot-based render layers
- renderer rebuilds that recreate render objects each frame

That is workable for playback, but it is a poor fit for:

- pointer hit testing
- stable selection state
- drag handles
- object inspection
- render-object refs or `userData`

The source-provenance id rule adds one more requirement:

- interaction must be traceable back to the authored Manim code that
  created the `Mobject`

## Current State

The current system behaves like this:

1. `Mobject` stores authoring state and methods.
2. scene evaluation produces flattened renderable snapshots
3. the WebGPU renderer converts snapshots into Three render objects
4. those render objects are destroyed and recreated on the next render

Current limitations:

- there is no stable retained `Object3D` per `Mobject`
- render-layer identity is not explicitly source-provenance-aware
- 3D authored objects are projected down before the WebGPU stage
- the stage component does not expose an interaction contract

## Non-Goals

This plan does not require:

- arbitrary direct mutation of `Object3D` transforms as the source of
  truth
- scene logic that depends on Three internals
- scene-specific hacks to fake interaction for one sweep
- breaking parity-oriented TS scene scripts to work around renderer gaps

This plan also does not preserve long-term support for a separate
non-Three.js TS Manim renderer.

## Design Principles

- one renderer for TS Manim, not parallel renderer stacks
- `Mobject` remains the semantic source of truth
- retained render objects mirror `Mobject` state
- interaction events resolve back to source `Mobject` ids
- object ids are stable across rerenders
- source-provenance ids are available in debug tooling and event payloads
- the first slice should work for 2D primitives before true 3D parity

## Target Architecture

Split the work into five layers:

0. Renderer convergence on Three.js
1. Authored identity and provenance
2. `Mobject` interaction metadata
3. Render snapshot identity propagation
4. Retained WebGPU/Three render objects
5. Pointer routing and hit testing

### 0. Renderer Convergence On Three.js

Before adding interaction, remove the requirement that TS Manim scenes
must keep working through a separate non-Three.js renderer.

Implications:

- the Three.js-backed renderer becomes the only TS Manim renderer
- SVG fallback stops being a compatibility target for new TS Manim work
- snapshot, picking, and retained-object design can target one runtime
- interaction features do not need duplicate implementations

Reason:

- stable interaction needs retained runtime objects
- retained runtime objects fit the Three.js path naturally
- preserving a second renderer would slow every interaction feature and
  force lowest-common-denominator API design

### 1. Authored Identity And Provenance

Add a formal source provenance model for `Mobject` creation.

Each interactive or renderable `Mobject` should carry metadata such as:

- `sourceFile`
- `sourceLine`
- `sourceColumn` if available
- `sourceIdSuffix` or explicit symbolic id
- `sourceEncodedId`

Rules:

- explicit ids must still encode file and line
- autogenerated ids must also encode file and line
- cloned objects must decide whether to preserve or derive provenance
- replacement/transform targets must remain distinguishable

Recommended normalized shape:

```ts
type MobjectSourceRef = {
  file: string;
  line: number;
  column?: number;
  label?: string;
};
```

And a derived stable id:

```ts
type EncodedMobjectId = string;
```

#### What Counts As An Authored Scene Module

For the current TS implementation, an authored scene module is a
repo-owned TypeScript file where a scene author directly writes Manim
scene code.

Current inclusion criteria:

- file path ends in `.ts`
- file path has no Vite query suffix like `?raw`
- file lives under
  `app/src/lib/ts-feature-sweep/ts/`
- or file lives under `app/src/lib/dlxn/scenes/`
- and the module imports tracked Manim factories from `$lib/manim` or
  `$lib/manim-api`

Current included examples:

- `app/src/lib/ts-feature-sweep/ts/mobjectsBasics.ts`
- `app/src/lib/ts-feature-sweep/ts/axesGraphsPlotting.ts`
- `app/src/lib/ts-feature-sweep/ts/geometryTextPrimitives.ts`
- `app/src/lib/dlxn/scenes/dlx3x2ThreeTiles.ts`

Current excluded examples:

- `app/src/lib/manim-api.ts`
- `app/src/routes/...`
- `app/src/lib/ts-feature-sweep/scenes/allScenes.ts`
- non-scene helper modules outside the authored scene directories

Reason for the current scope:

- provenance should reflect the line the scene author wrote
- wrapping scene-authoring call sites is more stable than patching ids
  later in the renderer
- limiting the transform to scene modules avoids rewriting unrelated app
  code

#### Where Phase 1 Is Exercised

The current phase 1 implementation is exercised in three places:

- build-time transformation of authored scene modules through the Vite
  pre-transform plugin
- runtime propagation in `Mobject` cloning APIs such as `copy()`,
  `generateTarget()`, and `always_redraw`
- debug and e2e verification through the TS scene route debug snapshot

Concrete exercised paths:

- authored scene sources under `app/src/lib/ts-feature-sweep/ts/`
- authored scene sources under `app/src/lib/dlxn/scenes/`
- runtime provenance attachment in `app/src/lib/manim-api.ts`
- debug exposure in
  `app/src/routes/ts-scenes/[script]/[scene]/+page.svelte`
- e2e helpers in `app/tests/e2e/helpers/ts-scene-debug.ts`

Concrete verification cases:

- `app/tests/e2e/mobjects-time-wrap.spec.ts` checks that `square` and
  `circle` now carry encoded ids and `sourceRef`
- `app/tests/e2e/axes-plotting.spec.ts` checks the same for `graph`
- the broader TS preview route checks in
  `gpu-sweep.spec.ts`, `images-svg-assets.spec.ts`,
  `rate-functions-timing.spec.ts`, and `ts-sweep.spec.ts` verify that
  the transform does not break scene loading

#### Vite Plugin Notes

This section documents the current addition to `app/vite.config.ts` for
someone who is not familiar with Vite plugins.

##### Plain-Language Model

Vite lets the app register plugin objects.

Each plugin can participate in specific parts of the module pipeline.
For this feature, the important idea is:

- Vite sees a source file before it is bundled
- a plugin can inspect that file
- a plugin can return modified source text
- the rest of the app then compiles the modified version

So the provenance feature works by rewriting scene modules during build
and dev startup rather than asking every scene author to call a helper
manually.

##### What Was Added To `vite.config.ts`

The config now registers two local plugins:

- `pySourcesPlugin()`
- `manimSourceIdPlugin()`

The config itself no longer holds the implementation details.
It only imports the plugin factories and passes them into the `plugins`
array.

Current plugin module locations:

- `app/vite-plugins/pySourcesPlugin.ts`
- `app/vite-plugins/manimSourceIdPlugin.ts`

##### What `manimSourceIdPlugin()` Does

This plugin runs as a Vite `transform` hook with `enforce: 'pre'`.

That means it runs early, before the rest of the normal compile
pipeline. Its job is:

1. check whether the current file is an authored scene module
2. parse the TypeScript source into an AST
3. collect tracked Manim factory names imported by that file
4. wrap matching constructor calls with
   `__attachMobjectSource(...)`
5. inject `import { __attachMobjectSource } from '$lib/manim-source'`
   if the file did not already import it

Example author code:

```ts
const square = Square('square', { size: 112, stroke: '#4CC9F0' });
```

Transformed shape:

```ts
const square = __attachMobjectSource(
  Square('square', { size: 112, stroke: '#4CC9F0' }),
  {
    file: 'app/src/lib/ts-feature-sweep/ts/mobjectsBasics.ts',
    line: 20,
    column: 18
  }
);
```

The runtime helper then stores:

- `mobject.sourceRef`
- an encoded `mobject.id` such as
  `app/src/lib/ts-feature-sweep/ts/mobjectsBasics.ts:20:square`

##### Why This Is In Vite Instead Of Runtime Stack Parsing

Build-time transformation is preferred because:

- it captures the exact authored file and line deterministically
- it avoids parsing JavaScript stack traces at runtime
- it works consistently in dev and build output
- it keeps scene code clean and close to ordinary Manim-style authoring

Runtime stack parsing would be slower and more fragile.

##### What The Plugin Does Not Do

The current plugin does not:

- rewrite every TypeScript file in the app
- infer provenance for arbitrary helper wrappers outside the authored
  scene directories
- change Python scene code
- replace the runtime clone-propagation logic in `manim-api.ts`

So the Vite plugin is only the call-site capture layer.
The runtime still needs to preserve provenance once an object has been
created.

### 2. `Mobject` Interaction Metadata

Add interaction behavior to `Mobject` as metadata, not as a new geometry
primitive.

Recommended additions:

- `interactive?: boolean`
- `draggable?: boolean`
- `cursor?: string`
- `userData?: Record<string, unknown>`
- `onPointerDown?: (...) => void`
- `onPointerMove?: (...) => void`
- `onPointerUp?: (...) => void`
- `onPointerEnter?: (...) => void`
- `onPointerLeave?: (...) => void`
- `bindRenderObject?: (object: Object3D | null) => void`

Important rule:

- interaction callbacks may observe the render object
- semantic scene changes should still be applied to the `Mobject`
  itself, then mirrored into rendering

### 3. Render Snapshot Identity Propagation

Extend the WebGPU snapshot layer types so each renderable layer keeps a
strong connection to its source `Mobject`.

Add fields such as:

- `mobjectId`
- `sourceFile`
- `sourceLine`
- `pickable`
- `cursor`
- `userData`

This data must survive:

- ordinary render passes
- intro animations
- replacement transforms
- morphing between source and target objects

### 4. Retained WebGPU/Three Render Objects

Change the WebGPU renderer from full rebuild mode to retained-object
mode.

Instead of recreating every render object each frame:

- keep a `Map<layerKey, Object3D>`
- update existing geometry/material/transform where possible
- remove only objects that disappeared from the snapshot
- attach id and provenance metadata to `object.userData`

This retained layer is the prerequisite for:

- stable picking
- hover enter/leave
- drag capture
- render-object refs
- debug inspection

### 5. Pointer Routing And Hit Testing

Add a pointer interaction controller at the stage level.

Responsibilities:

- listen to pointer events on the canvas container
- convert pointer position into stage coordinates
- hit test against retained render objects
- resolve the winning hit to a `Mobject`
- emit a typed interaction event payload

Recommended event payload:

```ts
type ManimPointerEvent = {
  mobject: Mobject;
  mobjectId: string;
  sourceFile: string;
  sourceLine: number;
  scenePoint: Point;
  nativeEvent: PointerEvent;
  object3d?: Object3D;
};
```

## Identity Rules

These rules are mandatory.

### Rule 1: Encoded Provenance

Every interactive renderable id must encode file and line from the
Manim source that created it.

### Rule 2: Repo-Relative Paths

Use repo-relative paths in the encoded id rather than absolute paths.

Reason:

- ids stay portable across machines
- debug output is shorter
- snapshot baselines remain stable

### Rule 3: Stable Rerender Identity

A rerender of the same authored object must preserve the same encoded
id.

### Rule 4: Derived Objects

If an object is derived from another one, its id should preserve the
origin and append a derivation suffix.

Examples:

- `py/01_scene.py:12:title`
- `py/01_scene.py:12:title__target`
- `py/01_scene.py:12:title__copy_1`

### Rule 5: Group Children

Child ids should preserve their own provenance rather than inheriting
only the parent group id.

### Rule 6: Debug Visibility

The encoded id and source provenance should be visible in:

- event payloads
- renderer debug logs
- browser inspection via `userData`
- optional future selection overlays

## API Shape

Recommended approach:

- add an `interactable(...)` helper or equivalent metadata helper
- avoid introducing geometry-specific wrappers such as
  `InteractiveCircle`

Example:

```ts
const node = interactable(
  Circle({
    id: "app/src/lib/scene.ts:42:node",
    radius: 0.5,
    color: BLUE
  }),
  {
    draggable: true,
    onPointerEnter: ({ mobject }) => {
      mobject.setFill?.("#38bdf8", 0.2);
    }
  }
);
```

## 2D First, Then 3D

The first milestone should target current 2D renderables:

- `square`
- `circle`
- `dot`
- `path`
- textured 2D overlays if practical

True Three-style 3D interaction should be a later phase.

Reason:

- today authored 3D objects are converted before WebGPU rendering
- true 3D picking needs native 3D render objects and raycasting
- that is a larger renderer-parity project

Even in the 2D-first slice, the runtime target is still the Three.js
renderer, not the legacy non-Three.js TS path.

## Migration Phases

### Phase 0: Drop Non-Three.js TS Manim Support

- make the Three.js-backed renderer the sole TS Manim renderer
- remove the requirement to preserve feature parity with the non-Three
  TS renderer
- collapse renderer-specific branching where practical
- treat remaining SVG-only behavior as migration debt, not as a
  permanent compatibility promise

Deliverable:

- one TS Manim renderer target for future interaction work

### Phase 1: Provenance Id Model

- add source provenance metadata to `Mobject`
- define canonical encoded id format
- update constructors and clone/copy/target flows
- document id derivation rules

Deliverable:

- all new renderables can expose file/line-derived ids

### Phase 2: Interaction Metadata In `Mobject`

- add interaction fields and event types
- keep API compatible with existing scene construction patterns
- ensure cloning preserves intended interaction metadata

Deliverable:

- authored scenes can mark objects interactive

### Phase 3: Snapshot Identity Propagation

- extend snapshot layer types with `mobjectId` and provenance fields
- preserve ids through transforms and replacements
- add tests for stable mapping

Deliverable:

- renderer receives enough identity data to support picking

### Phase 4: Retained Renderer Refactor

- replace full child teardown/rebuild with a retained object map
- sync transforms and materials in place
- attach provenance metadata to `userData`

Deliverable:

- stable runtime `Object3D` instances

### Phase 5: Pointer Controller

- add canvas-level pointer listeners
- implement hit testing for retained objects
- dispatch typed pointer events back to `Mobject`
- support hover enter/leave and pointer down/up

Deliverable:

- basic clickable/hoverable objects

### Phase 6: Dragging

- add pointer capture support
- map motion into scene-space movement
- update `Mobject` transforms, not raw render-object transforms
- add constraints if needed later

Deliverable:

- basic drag interaction for 2D primitives

### Phase 7: 3D Follow-Up

- stop flattening true 3D objects before the renderer
- add native 3D layer types
- use real 3D raycasting
- preserve the same provenance id contract

Deliverable:

- real Three-style interaction for 3D mobjects

## Validation

Validation should cover behavior, identity, and regressions.

### Unit Tests

- encoded id generation from file/line metadata
- clone and target id derivation rules
- snapshot layer identity propagation
- pointer-hit resolution to `mobjectId`

### Integration Tests

- hover enter/leave fires once per boundary crossing
- pointer down resolves the expected encoded id
- dragging updates the authored `Mobject`
- replacement transforms do not lose provenance ids

### Debug Checks

- inspect retained `Object3D.userData`
- confirm `mobjectId`, `sourceFile`, and `sourceLine`
- confirm rerenders preserve runtime object association

## Open Questions

- how should source provenance be captured for handwritten TS scenes:
  explicit constructor arg, helper macro, or callsite wrapper?
- should non-interactive but renderable objects also be required to
  encode file/line ids, or only interactive ones?
- what is the best derivation rule for copies and generated targets?
- should text and SVG overlays use DOM hit testing first or renderer-side
  picking from the start?

## Recommendation

Start with:

- provenance-aware ids
- retained renderer objects for 2D geometry
- pointer down and hover events

Do not start with:

- arbitrary direct `Object3D` mutation as public API
- full 3D picking
- scene-specific interaction hacks

That keeps the work aligned with the current Manim-like data model while
adding the minimum runtime layer needed for Three-style interaction.
